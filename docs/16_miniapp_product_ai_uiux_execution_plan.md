# 宠小护小程序产品、AI 与 UI/UX 执行优化计划

日期：2026-04-30  
角色视角：全栈工程师 + 产品经理 + 宠物照护顾问  
范围：`apps/miniapp`、`apps/api`、共享类型、产品路线与设计系统。

## 1. 当前阶段判断

宠小护现在处于 **0.7 版可演示 MVP 阶段**。

它已经不是空白原型：小程序、API、管理后台、AI 问答、健康记录、宠物档案、提醒、症状分诊、餐食建议、食物安全、就医摘要和趋势页都已经有骨架。但它也还没到可正式上线的 1.0，原因不是“页面数量不够”，而是 **用户主线、智能上下文、健康数据闭环和 UI 规范还没有稳定成一个整体产品**。

一句话判断：

> 当前能证明方向，但还不能稳定证明留存。下一阶段要从“功能入口集合”升级为“围绕一只宠物持续工作的日常照护助手”。

## 3. 核心问题

### 3.1 产品问题：主线仍然不够强

现在页面入口很多，但用户打开后仍可能问：

- 今天最重要的事情是什么？
- AI 是根据哪些记录判断的？
- 记录完之后有什么反馈？
- 症状分诊结果如何进入就医摘要？
- 提醒完成后是否沉淀到健康时间线？

理想主线应该是：

> 宠物档案 -> 今日计划 -> 快速记录 -> AI 洞察 -> 风险分诊 -> 就医摘要 -> 用户反馈 -> 知识库优化

当前这条链已经有节点，但节点之间还没有完全联动。

### 3.2 智能问题：AI 还像“带档案的问答”，不是“长期照护助理”

当前 AI 已经读取宠物档案和最近记录，这是很好的开始。但风险判断主要仍看用户当前问题，近期记录只进入回答文本，没有真正参与风险策略。

需要升级：

- 风险判断同时读取当前问题和最近健康记录。
- 症状分诊要从“选症状 + 一段描述”升级为“按症状动态追问关键问题”。
- AI 回答要明确说明“依据了哪些档案/记录/知识库”。
- 用户反馈按钮要真正提交后端，而不是只摆 UI。
- 健康记录要能反推首页 AI 洞察和风险提醒。

### 3.3 宠物专业问题：分诊结构还太粗

宠物健康类产品最危险的不是功能少，而是“看起来很聪明但漏掉急症”。目前红色急症已覆盖尿闭、呼吸困难、抽搐、误食毒物等，但仍需补：

- 呕吐：次数、颜色、是否带血、能否喝水、是否幼宠/老年宠、是否误食。
- 腹泻：便血/黑便、次数、是否换粮、是否伴随呕吐。
- 排尿异常：完全尿不出、频繁蹲、尿血、疼痛叫声。
- 呼吸异常：张口呼吸、舌色发紫、呼吸频率明显异常。
- 误食：食物/药物/植物名称、时间、摄入量、当前状态。
- 皮肤/耳眼：部位、范围、瘙痒、渗出、气味、是否扩散。

这类信息必须结构化进入 AI prompt 和就医摘要。

### 3.4 UI/UX 问题：页面已经有美化，但系统性不足

现状不是“完全没设计”，而是设计语言不稳定：

- 页面容器命名有 `.page-body`、`.page-content`、页面私有 padding，安全区容易不一致。
- 顶部自定义导航和页面横向 padding 在个别子页仍可能冲突。
- 底部导航是组件级模拟 tab，触控面积够用，但还需要更明确的选中态、无障碍语义和底部内容避让。
- 卡片、阴影、渐变较多，首页信息重量偏平均。
- 表单有 label，但错误反馈、追问结构和保存状态还不够。
- 结构性图标已经基本使用 SVG，方向正确，应继续避免 emoji 作为功能图标。

### 3.5 技术问题：数据模型出现重复和不一致

- `apps/miniapp/src/services/petStore.ts` 和 `apps/miniapp/src/stores/health.ts` 都有健康记录类型和存储逻辑，后者类型较旧，容易产生分叉。
- `createdAt` 目前用本地化字符串保存，后续排序、跨端解析、服务端同步会出问题，应该改为 ISO 字符串，显示时再格式化。
- 健康记录、任务、提醒仍以本地存储为主，适合 MVP，但上线前必须有云端用户、宠物、记录、提醒和对话表。

## 4. UI/UX 设计原则

结合本地 `ui-ux-pro-max` 输出、`ui-styling` 规则、WeUI 和 WCAG，宠小护适合的方向是：

> 温暖、可信、轻医疗、低焦虑、以每日任务和健康信息为中心。

本地 skill 推荐了偏 Claymorphism 的移动端风格，但由于宠小护涉及健康风险分诊，不建议走过度软糖、过度 3D、强装饰路线。可保留轻触感、柔和阴影和温暖色，但信息优先级必须清楚。

### 必须遵守

- 触控目标尽量不低于 44px/88rpx。
- 正文对比度优先满足 WCAG AA，普通文本至少 4.5:1。
- 每屏只突出一个主操作。
- 底部一级导航 4 项即可：首页/AI/记录/我的。
- 急症入口不放底部 tab，作为跨页高优先级入口。
- 固定底栏和输入栏必须给内容区预留底部空间。
- 表单必须有可见 label，复杂场景要提供追问和错误/空状态。
- 风险色只用于风险语义：红色急诊，黄色尽快咨询，绿色居家观察。

## 5. 信息架构建议

### Tab 1：今日

目标：每天打开就知道该做什么。

- 当前宠物状态。
- 下一项提醒。
- AI 今日洞察。
- 急症红灯入口。
- 今日计划。
- 快速记录。
- 健康摘要。

### Tab 2：AI

目标：像专家助理，不是普通聊天框。

- 当前宠物和上下文状态。
- 日常/症状/餐食模式。
- 快捷问题。
- 图片/视频入口先做“辅助观察预留”，不承诺诊断。
- 回答结构固定：风险等级、判断依据、可做事项、立即就医信号、就医准备、免责声明。
- 反馈按钮真实入库。

### Tab 3：记录

目标：让用户愿意每天轻量记录。

- 饮食、排泄、体征、异常四组快速记录。
- 时间线。
- 趋势入口。
- 异常记录优先级更高。
- 后续补图片记录。

### Tab 4：我的

目标：宠物资料和设置中心。

- 多宠物档案。
- 慢病、过敏、绝育、体重、饮食。
- 提醒设置。
- 数据导出。
- 隐私、免责声明、反馈。

## 6. 功能缺口优先级

### P0：必须补齐

1. 统一页面容器和底部安全区。
2. 让症状分诊有关键追问。
3. 风险判断读取最近健康记录。
4. AI 反馈按钮提交后端。
5. `createdAt` 改为 ISO 保存，显示层格式化。
6. 健康记录、首页、AI 使用同一份 `petStore` 数据。
7. 小程序真机检查：顶部胶囊、安全区、输入框和底栏遮挡。

### P1：做出差异化

1. AI 每日健康洞察由后端生成，而不是前端简单规则。
2. 就医摘要自动读取分诊答案、最近记录和图片。
3. 食物/毒物速查扩展到可搜索知识库。
4. 提醒完成后自动写入健康时间线。
5. 多宠物切换。
6. 图片记录和图片辅助观察。
7. 后台反馈审核转知识库草稿。

### P2：上线后增强

1. 云同步与微信登录。
2. 微信订阅消息。
3. RAG 向量检索。
4. OCR 病历/疫苗本。
5. 会员权益。
6. 家庭共享。
7. 兽医审核内容。

## 9. 验收标准

1. 用户打开首页 3 秒内知道下一件要做的事。
2. 快速记录 2 次点击内完成。
3. AI 回答能显示当前宠物和最近记录上下文。
4. 症状分诊至少收集持续时间、精神食欲、排尿排便、误食/照片等关键字段。
5. 最近记录出现红色信号时，后端风险判定不会被普通问题冲淡。
6. 底栏不遮挡内容，输入区不被底栏遮挡。
7. 反馈按钮能在后台形成待处理记录。
8. 文案不承诺诊断、治疗或处方。

## 10. 参考依据

- WeUI 小程序组件库强调与微信原生视觉体验一致，由微信官方设计团队和小程序团队为小程序设计，可作为小程序原生感参考：https://wechat-miniprogram.github.io/weui/docs/
- WeUI GitHub 说明其目标是改善并规范微信用户体验，包含 button、cell、dialog、toast、actionsheet、icon 等移动端常用组件：https://github.com/Tencent/weui
- WCAG 2.1 对普通文本最小对比度建议为 4.5:1，大号文本为 3:1：https://www.w3.org/WAI/WCAG21/Understanding/contrast-minimum.html
- WCAG 2.1 目标尺寸说明建议自定义触控目标至少 44 x 44 CSS pixels：https://www.w3.org/WAI/WCAG21/Understanding/target-size.html
- 本地 `ui-ux-pro-max` 设计系统检索结论：宠物健康 AI 产品适合“清洁、可信、可访问”的健康类视觉系统；Claymorphism 只适合轻量触感，不适合作为医疗风险界面的主导风格。
