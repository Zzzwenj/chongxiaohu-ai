# 知识库、RAG 与用户反馈闭环

## 1. 先回答你的问题

可以综合宠物医疗、喂养、行为、护理、用品、疾病分诊等知识，整理成我们自己的知识库，再让大模型每次回答前去知识库检索。这正是个人平台最应该做的方向。

但建议不要把知识直接“输入到大模型里”当成一次性训练，原因是：

- 成本高：微调或训练模型比 RAG 贵很多。
- 更新慢：宠物医疗、产品、法规、药品禁忌都要持续更新。
- 难审计：模型参数里为什么这么回答，很难追溯来源。
- 风险高：医疗类回答必须知道依据来自哪篇资料、何时审核、谁审核。

更推荐：

```text
宠物资料采集
  -> 清洗与结构化
  -> 切分知识片段
  -> 打标签
  -> 向量化
  -> 入库
  -> 用户提问时检索
  -> 大模型基于检索结果回答
  -> 用户反馈
  -> 人工/兽医审核
  -> 更新知识库
```

这就是 RAG：Retrieval-Augmented Generation，检索增强生成。

## 2. 知识库范围

第一阶段知识库分 8 类：

- 宠物疾病与症状分诊：呕吐、腹泻、尿闭、咳嗽、皮肤、耳朵、眼睛、误食。
- 日常喂养：主粮、湿粮、冻干、零食、饮水、换粮。
- 生命周期：幼宠、成年、老年、孕期、绝育后。
- 预防医疗：疫苗、驱虫、体检、绝育。
- 行为训练：乱尿、吠叫、抓咬、分离焦虑、猫砂盆问题。
- 日常护理：洗澡、剪指甲、刷牙、耳道、毛发。
- 就医准备：病情摘要、检查项目解释、化验单解释。
- 安全禁忌：人药、毒性食物、危险植物、误食处理。

## 3. 资料来源分级

知识库不能随便抓网页就用，要给来源分级。

### S 级

- 官方法规。
- 兽医教材。
- 兽医协会指南。
- 药品说明书。
- 大型动物医院公开科普且有医生署名的资料。

### A 级

- 知名宠物医疗机构文章。
- 大学动物医学资料。
- 品牌白皮书中的非广告性基础知识。
- 有执业兽医审核的内容。

### B 级

- 高质量养宠平台文章。
- 用户经验总结。
- 社群常见问题。

B 级内容只能作为“用户表达和场景补充”，不能作为疾病判断和用药依据。

## 4. 知识结构

不要只存一整篇 Markdown，要拆成可检索片段。

### knowledge_articles

保存文章级信息：

- 标题。
- 分类。
- 适用物种。
- 来源链接。
- 来源等级。
- 审核人。
- 更新时间。
- 是否发布。

### knowledge_chunks

保存检索片段：

- 片段正文。
- 适用物种。
- 症状标签。
- 疾病标签。
- 风险等级。
- 禁忌标签。
- 来源文章。
- 向量 embedding。

### safety_rules

保存硬规则：

- 红色急症触发词。
- 危险药物。
- 毒性食物。
- 禁止输出的处方行为。
- 必须提示就医的条件。

## 5. 回答流程

```text
用户提问
  -> 前端带上本次必要宠物信息
  -> 后端识别意图
  -> 安全规则先判断红色急症
  -> 关键词检索知识库
  -> 向量检索知识库
  -> 合并排序
  -> 取 Top 5-10 条知识片段
  -> 生成带引用依据的回答
  -> 规则二次校验
  -> 返回结果
  -> 用户反馈
```

关键点：红色急症不要完全依赖大模型，必须有规则兜底。

## 6. 用户数据本地优先

你说“每个用户的信息存在本地”，这个方向很好，尤其适合早期降低隐私和合规压力。

推荐策略：

- 宠物档案默认存在用户设备本地。
- 问 AI 时，只上传本次必要信息。
- 用户可以选择开启云同步。
- 云端保存 AI 对话时默认脱敏。
- 用户反馈进入知识库流程前，要去掉昵称、电话、地址、医院名等个人信息。

前端本地存储：

- 小程序：本地 storage + IndexedDB 类似能力视框架支持情况而定。
- H5：IndexedDB。
- 敏感字段：可做本地加密。

现实提醒：

- 本地数据可能丢失。
- 换手机无法自动同步。
- 小程序清缓存会影响数据。

所以更稳的是“本地优先，云同步可选”。

## 7. 用户反馈如何完善知识库

不要让用户反馈直接进入知识库，否则会污染答案。必须走审核流。

```text
用户点“回答有问题”
  -> 选择原因：不准确/太笼统/危险/不适用/缺少本地信息
  -> 用户补充说明
  -> 生成反馈工单
  -> 系统聚类同类问题
  -> 运营初审
  -> 查资料或请兽医审核
  -> 更新知识片段/安全规则/提示词
  -> 重新向量化
  -> 加入评测集
```

反馈状态：

- pending：待处理。
- triaged：已分类。
- needs_evidence：需要资料依据。
- needs_vet_review：需要专业审核。
- approved：可入库。
- rejected：不采用。
- published：已发布并重新索引。

## 8. Vue 技术实现建议

### 用户端

- uni-app + Vue 3。
- Pinia 管理本地宠物档案。
- 本地持久化保存档案和健康记录。
- 请求 AI 时组装最小必要上下文。

### 管理后台

- Vue 3 + Vite + Element Plus。
- 页面：
  - 知识文章管理。
  - 知识片段管理。
  - 用户反馈工单。
  - AI 对话抽检。
  - 红色急症规则。
  - 危险药物/食物规则。
  - 模型调用成本。

### 后端

- NestJS。
- PostgreSQL + pgvector。
- Redis。
- BullMQ 或轻量任务队列。
- 对象存储。

### AI 服务

- 单独做 `ai-orchestrator` 模块。
- 支持多模型供应商。
- 支持 RAG 检索。
- 支持回答后校验。
- 支持日志脱敏。

## 9. 最小可行版本

第一版只做这些就够：

- 人工录入 50 篇高频知识文章。
- 自动切分为知识片段。
- pgvector 向量检索。
- 20 条红色急症规则。
- 30 条危险食物/药物规则。
- 用户反馈按钮。
- 后台反馈工单。
- 每周人工更新知识库。

不要一开始做自动全网爬取。医疗健康类内容质量差异太大，自动爬取容易把错误知识带进系统。

## 10. 长期壁垒

这个平台真正会越做越强的资产是：

- 中文宠物场景知识库。
- 高质量症状分诊规则。
- 用户真实问题聚类。
- 宠物档案与健康记录结构。
- 兽医审核过的高风险内容。
- 回答错误案例和评测集。

大模型只是发动机，知识库和反馈闭环才是你的方向盘和刹车。
