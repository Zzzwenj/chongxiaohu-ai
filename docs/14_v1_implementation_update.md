# 宠小护 1.0 实施更新记录

日期：2026-04-29

## 本轮目标

把项目从“单一 AI 问诊页面”推进到“宠物档案 + 健康记录 + AI 个性化上下文”的 1.0 雏形。

## 已完成改造

### 1. AI 回答开始使用个人宠物上下文

小程序侧现在会把本地宠物档案、最近健康记录和健康摘要一起传给后端 AI 接口。

后端回答时会参考：

- 宠物名称、猫狗类型、年龄、体重、绝育、慢病等档案信息。
- 最近进食、饮水、便便、排尿、呕吐、体重、精神、皮肤等记录。
- 本地生成的健康摘要。

这样用户问“今天该喂多少”“吐了两次怎么办”“尿不出来怎么办”时，系统不再只看单句问题，而是能结合当前宠物背景给出更贴近个人情况的建议。

### 2. 健康记录能力扩展

健康页从基础 4 类记录扩展为 8 类高频记录：

- 进食
- 饮水
- 便便
- 体重
- 排尿
- 呕吐
- 精神
- 皮肤

这些记录后续会成为 AI 判断风险、生成就医摘要、做趋势分析的核心数据。

### 3. 首页快速记录形成闭环

首页浮动按钮里的快速记录不再只是展示入口，现在点击“进食 / 饮水 / 便便”会跳到健康页并自动打开对应记录弹层。

### 4. AI 页增加上下文提示

AI 助手页增加了“本次会参考”的提示，让用户知道系统会参考当前宠物档案和最近健康记录。

这对建立用户信任很重要：用户能感知到宠小护不是普通聊天机器人，而是在围绕自己的宠物持续积累状态。

### 5. 白屏风险复查

本轮构建检查通过：

- `pnpm typecheck`
- `pnpm build:miniapp`

微信小程序产物中首页、AI 页、健康页都已经生成 `usingComponents`，自定义组件能被微信识别。

### 6. 分诊、餐食和就医摘要打通档案

症状分诊页和餐食建议页现在都会带上本地宠物档案与最近健康记录请求 AI。

就医摘要页会自动读取宠物档案，并把最近健康记录放进医生可读摘要里，减少用户手动重复填写。

### 7. 提醒管理开始持久化

提醒管理页会读取本地已保存提醒，用户新增、编辑、删除、完成提醒后会继续保存在本地。

## 目前仍然缺少的关键能力

### 1. 图片和视频分析还没有真正接入

1.0 可以先放入口，但不要假装已经能医学级识别。真正要做需要：

- 前端上传图片/视频。
- 后端存储或临时转发媒体。
- 多模态模型识别。
- 识别结果进入同一套风险分诊规则。
- 明确提示“图片/视频只能辅助判断，不能替代线下检查”。

### 2. 知识库还偏静态

当前更像本地安全规则 + 基础知识命中。后续应该做：

- 宠物知识条目管理后台。
- 来源、更新时间、审核状态。
- 用户反馈“回答不准”后进入待审核池。
- RAG 检索和答案引用。

### 3. 提醒还不是完整日程系统

当前已经能本地保存提醒，但还需要：

- 自定义提醒。
- 疫苗/驱虫周期模板。
- 微信订阅消息。
- 过期提醒和完成记录。

### 4. 多宠物还不完整

现在本地只有默认单宠物档案。后续需要：

- 多宠物列表。
- 当前宠物切换。
- 每只宠物独立健康记录、提醒和 AI 上下文。

### 5. 就医摘要还需要更自动化

当前已经会带出档案和最近健康记录，但后续还应根据症状分诊结果自动生成更完整的结构化摘要：

- 主诉
- 症状时间线
- 饮食饮水变化
- 排便排尿变化
- 精神状态
- 已上传图片/视频
- AI 风险等级

## 下一步优先级

### P0：让 1.0 能稳定试用

1. 修复所有小程序真机兼容问题，重点检查顶部安全区、底部导航、返回按钮。
2. 把健康记录、提醒、AI 问答三条链路打通。
3. 增加 API 地址配置说明，避免微信开发者工具里反复出现本地请求失败。

### P1：增强智能感

1. AI 回答引用最近健康记录。
2. 症状分诊页提交后直接带着结构化信息进入 AI。
3. 餐食页根据体重、年龄、绝育、活动量生成更具体建议。
4. 就医摘要页自动生成医生可读版本。

### P2：准备商业化

1. 免费版限制每日 AI 次数。
2. 会员版开放更长健康历史、就医摘要、提醒模板。
3. 后续再考虑医院合作、用品推荐和保险导流。
